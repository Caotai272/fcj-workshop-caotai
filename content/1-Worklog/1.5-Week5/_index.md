# Week 5 Worklog

## Week 5 Objectives

In week 5, the main goal was to explore application and system security testing tools: **HCL AppScan, Nessus, and Burp Suite**. These are commonly used tools that help detect security vulnerabilities, assess risk, and check the safety level of applications or systems within an authorized scope.

HCL AppScan supports application security testing forms such as DAST, SAST, IAST, and open-source component testing; Nessus is a system vulnerability assessment tool; Burp Suite is a web application security testing toolkit developed by PortSwigger. ([HCL SW Blogs][1])

Key topics for the week include:

* Getting an overview of application and system security testing.
* Exploring the role of **HCL AppScan** in application security testing.
* Exploring the role of **Nessus** in scanning and assessing system vulnerabilities.
* Exploring the role of **Burp Suite** in request/response analysis and web application testing.
* Referencing the OWASP Web Security Testing Guide to understand the basic web security testing process.
* Noting the principle of only testing on authorized systems, lab environments, or demo applications.

## Week 5 Achievements

### Overview

This week, I explored security testing tools including **HCL AppScan, Nessus, and Burp Suite**. These tools have different scopes: AppScan focuses on application security testing, Nessus focuses on system vulnerability assessment, and Burp Suite supports web application security testing at the request/response level with manual analysis. ([HCL SW Blogs][1])

### Knowledge gained

After completing week 5, I have understood:

* The role of security testing in the system development and deployment process.
* That **HCL AppScan** can support application security testing through forms such as DAST, SAST, IAST, and open-source component testing. ([HCL SW Blogs][1])
* That **Nessus** is used to assess vulnerabilities, check configuration, and help detect weaknesses in a system. ([docs.tenable.com][2])
* That **Burp Suite** is a toolkit for web application security testing, including proxy, scanner, and request/response flow analysis functions. ([portswigger.net][3])
* How to reference the **OWASP Web Security Testing Guide** as a foundational document when learning the web application security testing process. ([OWASP Foundation][4])
* That security testing should only be performed on authorized systems, lab environments, or demo applications.

### Practice

During the learning and exploration process, I was able to:

* Learn the concepts of security testing, vulnerability assessment, and web application security testing.
* Explore the interface, purpose, and main feature groups of HCL AppScan.
* Explore Nessus and its role in scanning system vulnerabilities.
* Explore Burp Suite and how it supports request/response analysis in web applications.
* Reference the OWASP Web Security Testing Guide to understand common testing categories.
* Build a basic comparison table between HCL AppScan, Nessus, and Burp Suite.
* Note cautions about testing scope, authorized testing permission, and safety when using security tools.
